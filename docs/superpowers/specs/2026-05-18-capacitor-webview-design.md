# Design Doc: Capacitor WebView Workaround for Google Play Approval

**Date:** 2026-05-18
**Status:** Draft
**Topic:** Android WebView Implementation (Bypassing TWA Engagement Issues)

## 1. Goal
Create a functional Android WebView application using Capacitor that loads `https://divemap.gr`. This app will replace the current Trusted Web Activity (TWA) version during the Google Play closed testing phase to ensure engagement metrics (time spent, sessions) are correctly attributed to the app, thereby increasing the chance of production approval.

## 2. Success Criteria
- The app must support a "Test/Dev" version with a unique package ID (`gr.divemap.twa.dev`) to allow parallel installation with the production Play Store version.
- The production-bound version must use the package name `gr.divemap.twa`.
- The app must load `https://divemap.gr` immediately upon launch.
- The app must handle Android back button navigation within the WebView.
- The app must support Geolocation permissions (for the map feature).
- The app must be signable with the existing `android.keystore`.

## 3. Architecture
The implementation will use **Capacitor** integrated into the existing `frontend/` directory.

### Components:
- **Capacitor Core**: Manages the bridge between Web and Native.
- **Android Platform**: The native Android project generated by Capacitor.
- **Geolocation Plugin**: Handles the permission requests for GPS access.

## 4. Implementation Details

### 4.1 Setup
1. Install Capacitor dependencies in the `frontend/` directory:
   - `@capacitor/core`
   - `@capacitor/cli`
   - `@capacitor/android`
   - `@capacitor/geolocation`
2. Initialize Capacitor with:
   - Name: `Divemap`
   - Package ID: `gr.divemap.twa`
   - Web Dir: `dist` (or `build`, though we will primarily use the `server.url` config).

### 4.2 Configuration (`capacitor.config.ts`)
Set the `server` configuration to load the live site:
```typescript
{
  appId: 'gr.divemap.twa',
  appName: 'Divemap',
  webDir: 'dist',
  server: {
    url: 'https://divemap.gr',
    cleartext: true
  }
}
```

### 4.3 Native Android Adjustments
- **Permissions**: Update `frontend/android/app/src/main/AndroidManifest.xml` to include:
  - `android.permission.ACCESS_COARSE_LOCATION`
  - `android.permission.ACCESS_FINE_LOCATION`
- **Back Button**: Ensure the `MainActivity.java` (or Kotlin) is configured to handle the back button by going back in the WebView history instead of closing the app immediately.

## 5. Build Process
1. Run `npx cap sync android` to sync web configurations to the native project.
2. Build the app using Gradle:
   - `./gradlew assembleRelease` (for APK)
   - `./gradlew bundleRelease` (for AAB)
3. Signing will use the existing `/home/kargig/src/divemap/divemap-android/android.keystore`.

## 6. Risks and Mitigations
- **Risk**: Performance might be slightly lower than TWA.
  - **Mitigation**: This is a temporary version for approval. Once approved, the TWA version can be restored.
- **Risk**: Cache/Cookies might behave differently.
  - **Mitigation**: Capacitor WebView uses standard Chrome engine on Android; behavior should be nearly identical to TWA.
- **Risk**: Asset Links.
  - **Mitigation**: WebView apps don't strictly require `assetlinks.json` to hide the URL bar (the bar is never shown in a WebView), but it's good to keep it for future TWA compatibility.

## 7. Post-Approval Strategy
Once Google approves the app for Production:
1. Re-build the original TWA version using Bubblewrap.
2. Increment the `versionCode`.
3. Upload the TWA AAB as a standard update.
4. Google does not re-run the engagement check for subsequent updates once the app is in "Production" state.
